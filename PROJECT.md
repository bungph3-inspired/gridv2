## Safe-Edit Workflow (Edit-tool truncation mitigation)

The Edit tool in this Cowork mount has a chronic silent-truncation bug: writes get capped at the file's previous on-disk byte size, dropping the tail mid-statement. Root cause is the mount layer (Cowork's Linux mount on top of Windows), not the Edit tool itself â€” the tool sends correct bytes, the mount silently truncates. Bash heredoc + a single Python syscall bypasses the cap. Confirmed reproducible 2026-05-12; bit ~19 times across this project as of 2026-05-16.

**Default tool: `scripts/safe-edit.sh`.** Two modes:

```bash
# Replace one occurrence of old_str with new_str (errors if old_str isn't unique)
bash scripts/safe-edit.sh <file> --replace <old_file> <new_file>

# Wholesale rewrite (new files, restorations, whole-section replacements)
bash scripts/safe-edit.sh <file> --rewrite <new_file>
```

`<old_file>` / `<new_file>` are temp files Claude writes via heredoc (use `tmp/` in-project to avoid `/tmp/` permission collisions from prior sessions). The script reads the target via Python, performs the replace, atomic-renames the result back, and runs `node --check` on `.js`/`.cjs`/`.mjs` files. Single syscall on the write â€” bypasses the mount cap. Exits 1 on missing/non-unique old_str, 3 on syntax failure.

### Decision rule (strict)

The bug fires when an Edit **grows the file past its previous on-disk size**. Use this rule:

- **Edit grows the file (adds bytes)** â†’ **safe-edit, no exceptions.** This is the default. Includes any change where the new content is longer than the old content, in any file of any size. "It's only 5 lines" is not a reason to skip â€” today's incident log shows the cap firing on additions of ~25 bytes.
- **Edit shrinks or replaces in-place at same size** â†’ native Edit OK. Renaming a variable to a same-length name, swapping a constant value, deleting a line, fixing a typo that doesn't add chars. The cap doesn't fire when the file isn't growing.
- **Brand-new file** â†’ bash heredoc (`cat > file <<'EOF'`) or `safe-edit.sh --rewrite`. The first write of a new file via the Write tool works, but ANY subsequent Edit will cap at the initial size, so don't trust Write+Edit for growth.
- **Unsure if your change grows the file?** â†’ safe-edit. The few extra seconds are vastly cheaper than the ~10-minute recovery cycle a truncation costs.

### Verification (always)

- **After any batch of file changes**: `node verify/check_integrity.cjs` to confirm no silent truncation. The check walks the canonical file list and flags (a) byte count below per-file floor, (b) `node --check` syntax failures on .js/.cjs, (c) last non-whitespace line not ending in `} ; ) ] > */`.
- **End of any code-touching session**: `node verify/run_all.cjs` to run preflight + bundle + all jsdom suites. Currently 11 suites, ~11s end-to-end.
- **Restoring a truncated file**: `safe-edit.sh --replace` with the truncated tail as the anchor (the truncated last partial line is usually unique enough), OR `safe-edit.sh --rewrite` if reconstructing from context. Then `node --check`.

### Sandbox setup (one-time per session)

The project's node_modules was installed on Windows, so esbuild's native binary is win32-only and crashes inside the Cowork Linux sandbox. First time bundling in a new session: `npm install @esbuild/linux-x64 --no-save`. Adds the Linux binary alongside the Windows one without modifying package.json. Don't commit â€” sandbox-local.
